# Undo/Redo Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Coherent, native-feeling undo/redo for the input form — two independent mode-scoped stacks, Clear/Apply as single undoable steps, focus-follows-undo — per `docs/superpowers/specs/2026-07-10-undo-redo-design.md`.

**Architecture:** **Model-owned, per-mode `NSUndoManager`.** `AppModel` owns one `NSUndoManager` for Basic and one for Advanced and exposes the active one; the text views' editing undo, the form operations (Clear, Apply), and the Edit-menu Undo/Redo all target that single active manager. This deliberately replaces today's split between SwiftUI's `@Environment(\.undoManager)` (used by Clear) and the AppKit field-editor/window manager (used by text editing) — the split that caused the FIX-3 crash and the "Clear skipped by undo" bug.

**Tech Stack:** Swift 6.2 (app target is `MainActor`-isolated), SwiftUI + AppKit (`NSViewRepresentable` text views), `NSUndoManager`, Swift Testing.

## Global Constraints (from the spec — every task's requirements include these)

- **Scope:** form content only. Mode switching, the Presets pane, and the Settings pane are NOT undoable.
- **Two independent, mode-scoped stacks;** each persists across mode switches; undo never crosses modes.
- Within a mode: field edits **+** Clear Form **+** Apply Saved Values share **one** manager, ordered most-recent-first.
- **Hard constraint:** a mode's text views AND its form operations must register on the **same** mode-scoped manager. Do NOT use `@Environment(\.undoManager)` for form ops. (This is the FIX-3 lesson.)
- **Behavior:** native word-level text granularity; symmetric `⇧⌘Z` redo; session-only history (no persistence, no artificial cap); a new edit drops the redo stack.
- **Focus-follows-undo:** an undo/redo makes the affected field first responder and reveals the change; Clear/Apply undo focuses the first field of the mode.
- **Edit menu:** Undo/Redo reflect the active mode's stack (action-named labels; disabled when empty).
- Swift 6.2, `MainActor`; UI strings via `String(localized:)` in `UIStrings`. Commit trailer: `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.

- [ ] **Step 3: Implement**

In `AppModel`:
```swift
let basicUndo = UndoManager()
let advancedUndo = UndoManager()
var activeUndoManager: UndoManager { form.mode == .basic ? basicUndo : advancedUndo }

func clearActiveForm() {
    guard !form.activeIsEmpty else { return }
    let previous = form
    form.clearActive()
    scheduleEncode()
    registerFormUndo(restoring: previous, name: UIStrings.clearForm)
}

func apply(_ preset: Preset) {
    let previous = form
    form.apply(preset)
    scheduleEncode()
    // Register on the manager for the mode we LANDED in (preset's mode).
    registerFormUndo(restoring: previous, name: UIStrings.applySavedValues)
}

private func registerFormUndo(restoring previous: FormState, name: String) {
    let manager = activeUndoManager
    let current = form
    manager.registerUndo(withTarget: self) { target in
        target.form = previous
        target.scheduleEncode()
        target.registerFormUndo(restoring: current, name: name)  // symmetric redo
    }
    manager.setActionName(name)
}
```
Delete the old `undoManager:`-parameter versions of `clearActiveForm`/`restoreForm`. Add `UIStrings.applySavedValues = String(localized: "Apply Saved Values")`.

Note: registering `apply`'s undo on `activeUndoManager` after `form.apply` means it lands on the preset's mode's stack (apply sets `form.mode`), matching the spec ("target mode's stack").
